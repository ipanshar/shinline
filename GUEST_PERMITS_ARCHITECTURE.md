# Архитектура сервиса гостевых пропусков

## Проблема

Сейчас гостевой сценарий встроен в общий транспортный пропуск:

- В UI гостевой режим включается флажком внутри формы разрешения на въезд.
- Данные гостя хранятся прямо в `entry_permits`.
- Фактический въезд/выезд завязан на `visitors` и `entry_permit_id`, но домен гостя как отдельной сущности отсутствует.

Это уже создаёт смешение двух разных процессов:

1. транспортный пропуск на ТС;
2. заявка/разрешение на гостя как физическое лицо.

Если продолжить добавлять поля в `entry_permits`, быстро появятся проблемы:

- один пропуск начнёт описывать и человека, и машину, и встречающую сторону;
- будет трудно хранить гостя без ТС;
- будет трудно выдать гостю и человеку, и ТС отдельные пропуска, но в рамках одного визита;
- в дальнейшем будет тяжело строить журнал визитов, продления, повторные проходы и аналитику.

## Архитектурное решение

Нужен отдельный доменный модуль `Гостевые пропуска`, но не отдельная изолированная логика доступа.

Правильная граница такая:

- `GuestVisit` — корневая бизнес-сущность гостевого визита;
- `EntryPermit` — универсальный пропуск на проход/проезд;
- `Visitor` — факт въезда/выезда через КПП;
- `Truck` — ТС, если оно есть у гостя.

То есть гостевой сервис не должен заменять `EntryPermit`, он должен оркестрировать его.

## Целевая модель данных

### 1. Таблица `guest_visits`

Новая сущность заявки/визита гостя.

Рекомендуемые поля:

- `id`
- `yard_id`
- `status_id` или `workflow_status`
- `guest_full_name` обязательное
- `guest_iin` Не обязательное, индекс
- `guest_company_name` Не обязательное
- `guest_position` обязательное
- `guest_phone` обязательное
- `host_name` обязательное
- `host_phone` обязательное
- `visit_starts_at` обязательное
- `visit_ends_at` nullable для разового, обязательно для многоразового по политике бизнеса
- `permit_kind` enum: `one_time`, `multi_time`
- `has_vehicle` boolean
- `comment` nullable
- `created_by_user_id`
- `approved_by_user_id` nullable
- `cancelled_by_user_id` nullable
- `created_at`, `updated_at`

Дополнительно полезно сразу заложить:

- `last_entry_at`
- `last_exit_at`
- `closed_at`
- `source` (`operator`, `integration`, `import`)

### 2. Таблица `guest_visit_vehicles`

Одна заявка гостя может быть без ТС или с одним/несколькими ТС. Даже если сейчас нужен только один автомобиль, лучше сделать отдельную таблицу, чтобы не зашивать ограничение в схему.

Поля:

- `id`
- `guest_visit_id`
- `truck_id` nullable до момента создания/привязки ТС
- `plate_number` обязательное для снапшота
- `brand` nullable
- `model` nullable
- `color` nullable
- `comment` nullable
- `created_at`, `updated_at`

Почему не хранить только `truck_id`:

- номер может меняться;
- нужно сохранить исторический срез того, что было указано при оформлении;
- часть заявок может быть создана до появления ТС в справочнике.

### 3. Таблица связей `guest_visit_permits`

Связывает гостевой визит с пропусками, которые реально используются для прохода/проезда.

Поля:

- `id`
- `guest_visit_id`
- `entry_permit_id`
- `permit_subject_type` enum: `person`, `vehicle`
- `guest_visit_vehicle_id` nullable для `vehicle`
- `created_at`

Это позволит для одного визита иметь:

- один пропуск на человека;
- ноль или несколько пропусков на ТС.

### 4. Таблица `visitors`

Не заменять, а расширить:

- добавить `guest_visit_id` nullable;
- сохранить `entry_permit_id`;
- не ломать текущий DSS/KПП поток.

Так вы получите трассировку:

- заявка гостя;
- связанные пропуска;
- реальные события въезда/выезда.

## Что делать с текущими полями в `entry_permits`

Текущие поля `is_guest`, `guest_name`, `guest_company`, `guest_destination`, `guest_purpose`, `guest_phone` нужно считать переходными.

Решение:

1. не развивать эту схему дальше;
2. новые обязательные данные гостя хранить в `guest_visits`;
3. старые поля оставить на период миграции и обратной совместимости;
4. после перевода UI и API постепенно убрать их из основного сценария.

Иначе `entry_permits` окончательно превратится в смешанный контейнер без чёткой ответственности.

## Сервисный слой

Нужен отдельный сервисный модуль:

- `App\Services\GuestVisitService`
- `App\Services\GuestVisitPermitService`
- `App\Services\GuestVisitVehicleService`

Разделение ответственности:

### `GuestVisitService`

Отвечает за:

- создание заявки гостя;
- изменение статуса;
- валидацию бизнес-правил;
- закрытие визита;
- получение карточки гостя и списка гостей.

### `GuestVisitVehicleService`

Отвечает за:

- поиск или создание ТС в `trucks`;
- привязку ТС к гостевому визиту;
- проверку дубликатов номера;
- сохранение снапшота данных по ТС.

### `GuestVisitPermitService`

Отвечает за:

- создание одного пропуска на человека;
- создание отдельного пропуска на каждое ТС гостя;
- деактивацию разовых пропусков при завершении визита;
- синхронизацию транспортного пропуска с DSS через уже существующий `DssPermitVehicleService`.

Важно: гостевой сервис не должен напрямую копировать код из `VisitorsCotroller`. Логику создания `EntryPermit` лучше вынести в переиспользуемый application service, а старый controller перевести на него.

## Предлагаемая бизнес-логика

### Сценарий 1. Гость без ТС

1. Оператор создаёт `guest_visit`.
2. Создаётся пропуск типа `person`.
3. При проходе через КПП создаётся/обновляется запись `visitor` с `guest_visit_id`.
4. Если пропуск разовый, после выхода визит закрывается автоматически.

### Сценарий 2. Гость с ТС

1. Оператор создаёт `guest_visit`.
2. Если ТС нет в базе, сервис создаёт `truck`.
3. Создаётся пропуск `person` на гостя.
4. Создаётся отдельный пропуск `vehicle` на ТС.
5. Транспортный пропуск синхронизируется с DSS.
6. При выезде разовый транспортный пропуск снимается текущим механизмом.

### Сценарий 3. Многоразовый гостевой пропуск

1. `permit_kind = multi_time`.
2. Пропуски не закрываются после первого выхода.
3. Закрытие только вручную, по сроку действия или отдельной задачей планировщика.

## API-контракт

Не смешивать новый модуль с существующими ручками `/security/addpermit` и `/security/getpermits`.

Сделать отдельные endpoints:

- `POST /security/guest-visits/list`
- `POST /security/guest-visits/create`
- `POST /security/guest-visits/update`
- `POST /security/guest-visits/show`
- `POST /security/guest-visits/cancel`
- `POST /security/guest-visits/close`
- `POST /security/guest-visits/add-vehicle`
- `POST /security/guest-visits/remove-vehicle`

Дополнительно:

- `POST /security/guest-visits/{id}/issue-permits`
- `POST /security/guest-visits/{id}/revoke-permits`

Если REST-стиль не критичен, можно оставить POST-модель, чтобы не ломать стиль текущего проекта.

## Права доступа

Добавить отдельные permissions:

- `guest_visits.view`
- `guest_visits.create`
- `guest_visits.update`
- `guest_visits.close`
- `guest_visits.cancel`
- `guest_visits.issue_permits`

Не использовать `permits.view` как единственное право, иначе модуль гостей останется логически зависим от транспортных разрешений.

## Интеграция с DSS и КПП

Ключевой принцип:

- DSS должен знать только о транспортном пропуске на ТС;
- пропуск на человека хранится в приложении и участвует в операторском контуре;
- гостевой визит агрегирует обе сущности.

Это правильно, потому что DSS у вас уже работает вокруг `truck_id` и `EntryPermit`, а не вокруг физлица.

Сделать:

- новый page-компонент;
- новый менеджер гостей;
- новый подпункт `Гости` в операторской.

### Этап 4. Permit orchestration

Сделать:

- выпуск person permit;
- выпуск vehicle permit;
- синхронизацию vehicle permit с DSS;
- закрытие разовых пропусков при завершении визита.

### Этап 5. Миграция старых данных

Если нужно сохранить историю:

- перенести старые `entry_permits.is_guest = true` в `guest_visits`;
- создать связи на существующие `entry_permits`;
- старые страницы оставить в режиме read-only или скрыть гостевые записи из общего реестра разрешений.

## Что я рекомендую делать сейчас

Минимально правильный путь:

1. не расширять дальше `EntryPermitsManager`;
2. вводить новый модуль `Гости` как отдельный экран и отдельный backend-контур;
3. использовать `EntryPermit` как подчинённую сущность для выпуска пропусков;
4. связать гостевой модуль с `Visitor`, чтобы въезд/выезд и история были сквозными.

## Итоговое решение

Итоговая архитектура должна быть такой:

- `GuestVisit` отвечает за человека и бизнес-контекст визита;
- `Truck` отвечает за транспорт;
- `EntryPermit` отвечает за право прохода/проезда;
- `Visitor` отвечает за факт нахождения на территории;
- `DSS` работает только с транспортным пропуском.

Это даст нормальную масштабируемую модель без смешения доменов и без переписывания существующего DSS-контура.
